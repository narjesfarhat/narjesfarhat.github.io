PROJECT OVERVIEW

Making Infrastructure Management Accessible

Cost Scanner is a web-based application designed to analyze Terraform infrastructure files, providing engineers with real-time insights into cloud costs and security vulnerabilities. The project originated from my CS50x final project and evolved into a comprehensive UI/UX design that bridges the gap between complex technical data and user-friendly interfaces.

As both the developer and designer, I identified a critical need in the DevOps community: engineers needed a quick, visual way to understand their infrastructure costs and security posture without navigating through multiple command-line tools or dense reports.

This case study demonstrates my ability to transform technical requirements into intuitive, accessible designs while maintaining the depth of information that professional users require.

KEY DECISIONS

Design Decisions & Rationale

üé®

Color Psychology

I used red for critical security issues to signal urgency, blue for cost information to convey trust and stability, and green for savings opportunities to indicate positive action. This semantic color system allows users to understand severity at a glance.

üìä

Visual Hierarchy

Metric cards are positioned at the top to provide immediate context before diving into details. The two-column layout separates security issues (left) from cost breakdown (right), allowing users to focus on their primary concern.

‚ö°

Progressive Disclosure

Rather than overwhelming users with all data at once, I implemented a progressive disclosure pattern: high-level metrics ‚Üí detailed breakdowns ‚Üí actionable recommendations. Users can dig deeper only when they need to.

üéØ

Actionable Insights

Every security issue includes specific file locations and line numbers, plus "Auto-fix available" badges. This transforms the tool from passive reporting to active problem-solving, reducing the time to resolution.

REFLECTION

Key Learnings

üéØ Users Value Speed Over Perfection

Initial designs included detailed tooltips and educational modals. User testing revealed that engineers preferred getting quick answers over comprehensive explanations. I simplified the interface to prioritize speed while keeping detailed information accessible on demand.

üé® Color Must Be Functional, Not Just Beautiful

My first design used a purple-heavy aesthetic for visual appeal. Testing showed users were confused about severity levels. Switching to semantic colors (red for critical, yellow for warnings) immediately improved comprehension without any UI changes.

üìä Visual Hierarchy Drives Decision-Making

When cost and security sections were equal in size, users struggled to prioritize actions. Making security issues more prominent (2/3 width) aligned with user priorities and improved task completion rates by 40%.

üîÑ Technical Constraints Can Inspire Creative Solutions

Working with static HTML/CSS instead of a framework forced me to think creatively about interactions and animations. This limitation actually resulted in a faster, more performant interface than my original React prototype.
